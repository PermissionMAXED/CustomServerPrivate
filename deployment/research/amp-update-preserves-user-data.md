# AMP Update — How To Preserve User Data While Updating Game Binaries

Source survey: `CubeCoders/AMPTemplates@main` (cloned 2026-05-28), Discourse knowledge base
posts, and the BAP CustomServer template in this workspace.

User requirement, paraphrased: *"If only the game files change, the update must touch only
those game files. Player stats, inventories, admin state, etc. must never be overwritten by
an Update."*

## 1. Short answer

AMP does **not** have a per-stage `ExcludePaths` / `PreserveFiles` flag, and `.backupExclude`
files are unrelated to updates (they are only consulted by the *Backup* feature). The
data-preservation behaviour you see in shipped templates (Minecraft, Valheim, ARK, Project
Zomboid, Factorio, Terraria) is achieved by combining four mechanisms:

1. Putting user data in a directory the update never touches.
2. Per-stage `"OverwriteExistingFiles": false` so default configs are dropped only when
   missing.
3. An explicit *Backup → Update → Restore* dance in `updates.json` for the rare files that
   the upstream archive **does** ship and would otherwise clobber.
4. Update mechanisms (SteamCMD, partial archive extracts) that by design only touch files
   they explicitly know about and never delete unmanaged files.

`.backupExclude` is the closest thing to a "shipped exclusion file", but it controls
*Backups*, not *Updates*. The only `Stash`-style mechanism in AMP is `git stash` against
the *template* repo when refreshing remote sources; it has nothing to do with instance
files.

## 3. The four real mechanisms, with concrete examples

### 3.1 Mechanism A — Put user data outside the update target

This is the foundation. **An update only writes files it explicitly fetches**. A SteamCMD
stage only writes Steam-manifest files. A `FetchURL` + `UnzipUpdateSource` stage only
writes the contents of the zip. If your user-data path is not produced by either, the
update never sees it.

Conventions used by official templates:

| Game | Update target (`App.BaseDirectory`) | User data lives at |
|---|---|---|
| ARK SA | `./ark-sa/2430930/` | `2430930/ShooterGame/Saved/` (never in the SteamCMD manifest) |
| Project Zomboid | `./project-zomboid/380870/` | `./project-zomboid/Zomboid/` (sibling, outside `380870`) |
| Valheim | `./Valheim/896660/` | `$HOME/.config/unity3d/IronGate/Valheim/worlds_local/` (entirely outside the instance) |
| Factorio | `./factorio/server/` | `./factorio/saves/` (sibling of binaries) |
| Minecraft Java | `./minecraft/` | `./minecraft/world/` (the `server.jar` swap leaves the world dir alone) |

This is design discipline. AMP does not enforce it; the template author chooses paths so
that the update's footprint and the user-data footprint do not overlap.

**Recommendation for the BAP custom server.** Your state files already live under
`data/` (`admin-state.json`, `economy-state.json`, `ranked-state.json`, `shop-state.json`,
`friends-state.json`, `players/`) and audit logs under `logs/`. Two adjustments make this
mechanism alone airtight:

- Make sure `Build-AmpFullLinuxWinePackage.ps1` keeps deleting `data/*-state.json` and
  `data/players/*` and `logs/*` from the staging dir before zipping. It already does this
  for `admin-state.json`, `economy-state.json`, `ranked-state.json` (line 154–156). Extend
  that list to **every** runtime state file: `shop-state.json`, `friends-state.json`,
  `match-history.jsonl`, the `players/` directory, and `logs/admin-audit.jsonl`.
- Make sure `dotnet publish` does not pull `data/asset-index.json` into the publish dir if
  that asset reference is regenerated by the operator. (Either leave it in, because it is
  an asset reference and not user state, or move it under a separate `assets/` path.)

If the zip never contains `data/*-state.json`, the AMP update — which is just
`extract zip over base dir with OverwriteExistingFiles=true` — physically cannot rewrite
those files, because there is nothing in the archive at that path.

### 3.2 Mechanism B — `"OverwriteExistingFiles": false` per UpdateStage

This is the closest thing AMP has to a "skip if user has customised this" flag. It applies
to a single UpdateStage, not globally. It works for `FetchURL`, `CopyFilePath`,
`CreateFile`, `CreateDirectory`, `ExtractArchive`, and `UnzipUpdateSource`. Behaviour:

- `OverwriteExistingFiles: true` — replace the destination unconditionally. Used for
  game-binary downloads.
- `OverwriteExistingFiles: false` — write only if the destination does not exist. Used
  for default configs, default save dirs, default permission files.

Examples from official templates:

`ark-saupdates.json`: ship a default `Game.ini` *only* if there isn't one yet:
```json
{
  "UpdateStageName": "Game.ini Config Download",
  "UpdateSource": "FetchURL",
  "UpdateSourceData": "https://raw.githubusercontent.com/CubeCoders/AMPTemplates/main/ark-sagame.ini",
  "UpdateSourceArgs": "Game.ini",
  "UpdateSourceTarget": "{{$FullBaseDir}}ShooterGame/Saved/Config/WindowsServer/",
  "OverwriteExistingFiles": false
}
```

`factorioupdates.json`: derive each user config from the bundled `*.example.json`, but
keep an existing one untouched:
```json
{
  "UpdateStageName": "Server-Settings File Copy",
  "UpdateSource": "CopyFilePath",
  "UpdateSourceData": "{{$FullBaseDir}}data/server-settings.json",
  "UpdateSourceArgs": "{{$FullBaseDir}}data/server-settings.example.json",
  "OverwriteExistingFiles": false
}
```

Plus `CreateFile`/`CreateDirectory` with `OverwriteExistingFiles: false` to seed empty
admin/whitelist/banlist files exactly once.

`project-zomboidupdates.json`: ensure `mods/` exists without ever wiping it:
```json
{
  "UpdateStageName": "Mods Directory Creation",
  "UpdateSource": "CreateDirectory",
  "UpdateSourceArgs": "{{$FullBaseDir}}Zomboid/mods",
  "OverwriteExistingFiles": false
}
```

**Recommendation for the BAP custom server.** Add explicit "seed if missing" stages for
your config and data directories:

```json
{
  "UpdateStageName": "Create state directory",
  "UpdateSourcePlatform": "All",
  "UpdateSource": "CreateDirectory",
  "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/data",
  "OverwriteExistingFiles": false
},
{
  "UpdateStageName": "Create players directory",
  "UpdateSourcePlatform": "All",
  "UpdateSource": "CreateDirectory",
  "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/data/players",
  "OverwriteExistingFiles": false
},
{
  "UpdateStageName": "Create logs directory",
  "UpdateSourcePlatform": "All",
  "UpdateSource": "CreateDirectory",
  "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/logs",
  "OverwriteExistingFiles": false
}
```

`appsettings.json` is the only config file you currently ship inside the zip, and the
operator may have edited the AMP-generated copy. Either rename your packaged copy to
`appsettings.example.json` and add a `CopyFilePath` stage with
`OverwriteExistingFiles: false`, or move every operator-tunable value out of
`appsettings.json` and into AMP `App.AppSettings` so the file does not need to ship.

### 3.3 Mechanism C — Backup → Update → Restore (the `.bak` dance)

When the upstream archive forces a default file into a path you must preserve (you cannot
move the file and you cannot stop the archive from writing it), the template stages an
explicit copy aside before the destructive stage and a restore after.

Pattern (verbatim from `minecraft-bedrockupdates.json`):

```json
{ "UpdateStageName": "Allowlist.json Backup",
  "UpdateSource": "CopyFilePath",
  "UpdateSourceData": "{{$FullBaseDir}}allowlist.json.bak",
  "UpdateSourceArgs": "{{$FullBaseDir}}allowlist.json",
  "OverwriteExistingFiles": true,
  "SkipOnFailure": true },

{ "UpdateStageName": "Server Download",
  "UpdateSource": "FetchURL",
  "UpdateSourceData": "https://www.minecraft.net/.../bedrock-server-...zip",
  "UnzipUpdateSource": true,
  "OverwriteExistingFiles": true,
  "DeleteAfterExtract": true },

{ "UpdateStageName": "Allowlist.json Restore",
  "UpdateSource": "CopyFilePath",
  "UpdateSourceData": "{{$FullBaseDir}}allowlist.json",
  "UpdateSourceArgs": "{{$FullBaseDir}}allowlist.json.bak",
  "OverwriteExistingFiles": true,
  "SkipOnFailure": true }
```

ARK SA does the same thing for the ServerAPI Permissions plugin's `config.json`.

Notes on the syntax that are not obvious:

- `UpdateSourceData` is the **destination**, `UpdateSourceArgs` is the **source** for
  `CopyFilePath` (this is reversed from intuition; verify against
  `ark-saupdates.json` lines 30–48 and `minecraft-bedrockupdates.json` first/last stages).
- `SkipOnFailure: true` on the backup stage handles the first-ever update where the file
  to back up does not yet exist.
- The `.bak` lives inside the instance dir, so it survives between updates but is also
  inside the user-data scope you want to preserve.

**Recommendation for the BAP custom server.** Use this only for files your packager
*cannot* exclude from the zip. Right now everything user-mutable is under
`BapCustomServer/data/`, `BapCustomServer/logs/`, and `Mods/BapCustomServer.ini`. The mod
ini lives under `game/Mods/BapCustomServer.ini` and is currently overwritten on every
update because the package zip ships a templated copy. Two options:

1. Strip `Mods/BapCustomServer.ini` from the zip and seed it via a separate
   `OverwriteExistingFiles: false` stage. Cleanest.
2. Keep shipping a fresh `BapCustomServer.ini` but add a Backup→Update→Restore for it,
   merging only the operator-controlled lines back. More fragile.

Option 1 is what Minecraft Bedrock would do if `allowlist.json` were not part of the zip;
they only need the dance because Mojang's archive contains `allowlist.json` and
`permissions.json` and they cannot remove them.

### 3.4 Mechanism D — Use `SteamCMD` / archive semantics that skip unmanaged files

`UpdateSource: SteamCMD` only ever writes files Steam knows about. User-created files
under the install dir survive the update. This is why
`App.BaseDirectory=./ark-sa/2430930/` is safe even though saves live at
`2430930/ShooterGame/Saved/SavedArks/`.

`UnzipUpdateSource: true` (zip extract) **overwrites files in the archive**, but it does
**not** delete files that are not in the archive. This is the same property that lets you
ship a newer game binary without erasing user data the operator has placed alongside it.

If you ever switch from a flat zip extraction to something like
`rm -rf BapCustomServer && extract zip` you would defeat this entirely. The current
`updates.json` is correct: a single zip extract with `OverwriteExistingFiles: true` does
the right thing **as long as the zip contains only files you intend to overwrite**.

## 4. Things to confirm in the BAP packaging script

Run a one-shot audit against `bapcustomserver-amp-full-linux-wine.zip` to confirm none of
the following paths are inside the zip:

```text
BapCustomServer/data/admin-state.json
BapCustomServer/data/economy-state.json
BapCustomServer/data/ranked-state.json
BapCustomServer/data/shop-state.json
BapCustomServer/data/friends-state.json
BapCustomServer/data/players/*
BapCustomServer/logs/*
BapCustomServer/data/match-history.jsonl
```

If any of these are present in the zip, an Update will overwrite live operator data
unconditionally. The current `Build-AmpFullLinuxWinePackage.ps1` only deletes three of
them; extend the deletion list.

For the in-game side, `Mods/BapCustomServer.ini` is currently re-templated every update
(packager rewrites `Host=`, `Port=`, `UseLocalProxy=`). Decide whether the operator should
ever be allowed to hand-edit it on the instance: if yes, drop it from the zip and seed via
`OverwriteExistingFiles: false`. If no, leave the current behaviour and document it.

## 5. Recommended `updates.json` structure for BAP

Take what you have today and layer the four mechanisms in this order:

```json
[
  {
    "UpdateStageName": "Backup operator-mutable config (if present)",
    "UpdateSource": "CopyFilePath",
    "UpdateSourceData": "{{$FullBaseDir}}BapCustomServer/appsettings.json.bak",
    "UpdateSourceArgs":  "{{$FullBaseDir}}BapCustomServer/appsettings.json",
    "OverwriteExistingFiles": true,
    "SkipOnFailure": true
  },
  {
    "UpdateStageName": "Download BAPBAP Custom Server full package",
    "UpdateSource": "GithubRelease",
    "UpdateSourceArgs": "Sonic0810/BAPBAP-CustomServer-AMPTemplates",
    "UpdateSourceData": "bapcustomserver-amp-full-linux-wine.zip",
    "UpdateSourceTarget": "{{$FullBaseDir}}",
    "UnzipUpdateSource": true,
    "OverwriteExistingFiles": true,
    "DeleteAfterExtract": true,
    "SkipOnFailure": false
  },
  {
    "UpdateStageName": "Restore operator-mutable config",
    "UpdateSource": "CopyFilePath",
    "UpdateSourceData": "{{$FullBaseDir}}BapCustomServer/appsettings.json",
    "UpdateSourceArgs":  "{{$FullBaseDir}}BapCustomServer/appsettings.json.bak",
    "OverwriteExistingFiles": true,
    "SkipOnFailure": true
  },
  {
    "UpdateStageName": "Seed data directory",
    "UpdateSource": "CreateDirectory",
    "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/data",
    "OverwriteExistingFiles": false
  },
  {
    "UpdateStageName": "Seed players directory",
    "UpdateSource": "CreateDirectory",
    "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/data/players",
    "OverwriteExistingFiles": false
  },
  {
    "UpdateStageName": "Seed logs directory",
    "UpdateSource": "CreateDirectory",
    "UpdateSourceArgs": "{{$FullBaseDir}}BapCustomServer/logs",
    "OverwriteExistingFiles": false
  },
  /* SetExecutableFlag stages stay as-is */
]
```

Combined with a packager that strips every `data/*-state.json`, `data/players/*`,
`logs/*`, and `match-history.jsonl` from the zip, an AMP **Update** will:

- replace every game binary, every server DLL, every Wine helper, every static asset;
- preserve every player stat, inventory, ranked entry, economy balance, shop record, ban,
  admin grant, audit-log entry, and match-history line, because none of those file paths
  are inside the update zip;
- preserve `appsettings.json` because the dance copies it aside and back;
- never wipe `data/`, `data/players/`, or `logs/` because they are seeded only when
  missing.

## 6. Optional — backup hygiene

For the *Backup* feature (separate from Update), you can additionally ship a
`.backupExclude` file at the BAP root listing game binaries you do not need to back up
(everything under `game/`, `MelonLoader/`, etc.). That makes operator backups small and
fast and is exactly what `exclusions/.backupExclusionsArkSA1` does for ARK. It does not
help with Update preservation, but it complements the strategy: small data-only backups +
non-destructive updates = if anything ever does go wrong on update, the operator restores
just the data backup over the freshly updated binaries.

## 7. What does *not* exist (so do not look for it)

- No `ExcludePaths` / `PreservePaths` / `RetainFiles` field on UpdateStages. Searched the
  full template repo: no occurrences.
- No global "smart-exclude" file consulted by the updater. `.backupExclude` is consulted
  only by the Backup feature (Discourse post 2282).
- No `SkipIfFileExists` flag. The equivalent is `OverwriteExistingFiles: false` on the
  stage that would otherwise write the file.
- No `App.UpdateExcludePaths` kvp key. Update behaviour is fully determined by what stages
  do, what files they touch, and `OverwriteExistingFiles`.
- No `git stash`-style instance-file stash. The "No stash entries found" line in AMP logs
  refers to the `git stash` command AMP runs against the *template* repository while
  refreshing remote sources; it has nothing to do with instance files. It is not a
  failure.
